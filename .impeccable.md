## Design Context

### Users
Plembfin is for self-hosters who run Plex, Emby, Jellyfin, and related media tools on their own infrastructure. They use it to monitor playback, browse movie and TV history, sync watched state and resume progress, inspect metadata, manage integrations, review stats, and keep their local media stack accurate across services.

Users are comfortable with operational tools and expect the interface to be efficient, predictable, and information-dense. They are often checking state, comparing records, diagnosing sync issues, or making small corrections rather than casually browsing a marketing-style experience.

### Brand Personality
The product should feel simple, detailed, and accurate.

The tone is calm and utilitarian: clear labels, direct feedback, and controls that expose enough detail without making common workflows feel heavy. Confidence comes from precision, consistency, and visible system state rather than decorative flourish.

### Aesthetic Direction
The current Plembfin styling is the source of truth and should be preserved when adding or changing features.

The interface is a self-hosted media operations dashboard with a media-library layer: dark-first, compact, poster-rich, and structured around a persistent left sidebar plus a shared page topbar. Light mode exists and should remain polished, but new work should fit the established dark-first visual language.

Use the existing visual system:
- Existing logo assets, sidebar layout, topbar controls, panel surfaces, poster grids, list rows, detail pages, and settings patterns.
- Existing `public/styles.css` tokens for color, spacing, radius, typography, and control sizing.
- Existing compact control treatments for search, sort, filters, toggles, sliders, icon buttons, action pills, and topbar actions.
- Existing media-first imagery: posters, backdrops, logos, cast photos, and app icons should carry the visual interest where relevant.

Poster styling is standardized: all poster artwork and poster placeholders should use the shared `--poster-radius` token from `public/styles.css` (currently 8px), matching the TV Shows library poster corners. Do not introduce one-off poster corner radii for movie posters, show posters, history posters, upcoming posters, stats posters, part-watched posters, detail posters, or poster fallbacks. Profile photos, cast headshots, logos, backdrops, and non-poster thumbnails may use their own shape when the existing UI already does so.

Do not introduce a new brand direction, new decorative motif, marketing-page composition, oversized hero sections, gradient-heavy styling, ornamental backgrounds, or unrelated design-system conventions. Changes should look like they were already part of Plembfin.

### Design Principles
1. Preserve the current UI language.
   Extend existing components and CSS patterns before inventing new ones. New pages and controls should align with the current sidebar, topbar, dashboard, explorer, detail, stats, settings, and modal patterns.

2. Prioritize operational clarity.
   Make status, sync state, search scope, dates, availability, and actions easy to scan. Prefer precise labels and stable layouts over decorative copy or visual novelty.

3. Keep density useful, not crowded.
   Plembfin is an admin-style media tool, so compact layouts are appropriate. Use consistent spacing, alignment, and responsive constraints so controls and text remain readable without wasting screen space.

4. Let media artwork provide richness.
   Posters, backdrops, logos, and profile images should be the primary visual assets. Avoid generic illustrations or decorative elements that compete with actual media artwork.

5. Keep poster treatment consistent.
   Poster artwork across libraries, dashboard rows, detail pages, stats, history, upcoming, dialogs, and placeholders uses `--poster-radius`. If a new poster surface is added, wire it to the token instead of hard-coding a radius.

6. Design every change for desktop and mobile.
   Desktop should support fast scanning and comparison. Mobile should adapt workflows without hiding critical controls, using the existing responsive topbar, sidebar, calendar, dashboard, and grid/list patterns.

7. Maintain accuracy and trust.
   Empty states, loading states, errors, badges, and metadata should communicate exactly what is known, unknown, cached, missing, synced, or actionable.
